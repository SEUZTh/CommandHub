# CommandHub V1.2 计划

## 摘要

- 将命令历史从内存数组切换为本地 SQLite 持久化，保留 `usage_count`、`last_used_at`、`created_at`，应用重启后数据不丢。
- 搜索改为“两阶段”方案：SQLite 先做候选粗筛，再在内存做 fuzzy + 综合打分，避免每次查询都 `fetchAll()`。
- 排序保留 `匹配度 + 频率 + 最近使用` 的核心思路，但 `usageCount` 改为对数归一化，避免高频命令永久霸榜。
- `CommandContext` 不从链路中移除；v1.2 保留接口并预留数据库字段，当前不参与排序和 UI。
- Launcher 每次打开时重置搜索状态，重新拉取结果并聚焦输入框，不依赖 `onAppear` 做状态初始化。

## 关键变更

### 存储与数据模型

- 新增 `DatabaseManager`，负责：
  - 在 `Application Support/CommandHub/commandhub.sqlite` 打开数据库并确保目录存在。
  - 应用启动时创建表、索引，并持有单个 SQLite 连接。
  - 通过专用串行队列包装全部 SQLite 调用，避免剪贴板监听、搜索、选择回写并发访问时出错。
- `commands` 表在原字段基础上增加一个上下文预留列：
  - `id TEXT PRIMARY KEY`
  - `command TEXT NOT NULL`
  - `usage_count INTEGER DEFAULT 0`
  - `last_used_at INTEGER`
  - `created_at INTEGER NOT NULL`
  - `source TEXT`
- 保留 `CREATE INDEX idx_command ON commands(command);`
- `CommandItem` 用于 Launcher 展示与搜索评分，字段为：
  - `id: String`
  - `command: String`
  - `usageCount: Int`
  - `lastUsedAt: TimeInterval?`
  - `createdAt: TimeInterval`
- `CommandContext` 类型保留；`StorageService` 接口保持为：
  - `save(command: String, context: CommandContext? = nil)`
  - `fetchCandidates(query: String, limit: Int = 200) -> [CommandItem]`
  - `markUsed(id: String)`
  - `delete(id: String)`
- 去重规则维持“全历史精确去重”：
  - 以 Parser 输出后的完整字符串为准。
  - 命中已有完全相同 `command` 时直接跳过，不更新已有记录。
  - `normalizedCommand`、alias 合并留到后续版本。

### 搜索与排序

- 删除“每次搜索都 `fetchAll()`”的方案，改为候选集查询：
  - 空 query：只取前 `200` 条高价值命令，按 `usage_count DESC, last_used_at DESC, created_at DESC` 取候选。
  - 非空 query：构造 subsequence `LIKE` 模式作为粗筛，例如 `kctl -> %k%c%t%l%`，先在 SQLite 中取最多 `200` 条候选，再进入内存评分。
- 选择这种粗筛方式而不是简单 `contains(query)`，原因是必须保留 `kctl -> kubectl`、`gco -> git checkout` 这类缩写 fuzzy 能力。
- `FuzzyMatcher.matchScore(query:target:)` 继续使用轻量 subsequence 匹配：
  - query 为空返回 `1.0`
  - 无法按顺序命中返回 `0`
  - 否则返回 `matchedCount / target.count`
- `CommandScorer.score(item:query:)` 改为归一化打分：
  - `matchScore * 0.4`
  - `usageScore * 0.4`
  - `recencyScore * 0.2`
- `usageScore` 使用对数增长并截断到 `0...1`：
  - `usageScore = min(1.0, log1p(Double(item.usageCount)) / 5.0)`
- `recencyScore` 继续使用 7 天线性衰减：
  - 最近 7 天内线性从 `1` 衰减到 `0`
  - 从未使用过的命令为 `0`
- 最终排序固定为：
  - `score` 降序
  - `usageCount` 降序
  - `lastUsedAt` 降序（`nil` 在后）
  - `createdAt` 降序
  - `command` 升序
- 搜索结果只展示排序后的候选集，不再默认拉取全库。

### 导入链路与 Launcher 接入

- 剪贴板导入链路保持为：
  - `copy -> parse -> resolveContext -> save(SQLite)`
- `ClipboardService` 继续调用 `ContextResolver.resolve()` 并传给 `StorageService.save(command:context:)`。
- v1.2 的 `source` 字段仅做预留：
  - 当前可为空。
  - 若后续 `ContextResolver` 能拿到真实来源，可直接写入，无需再改 schema。
  - 搜索、排序、UI 目前忽略该字段。
- `LauncherViewModel` 改为持有结果状态，而不是直接读内存数组：
  - `@Published var query`
  - `@Published var results: [CommandItem]`
  - `@Published var selection: String?`
  - `activate()`
  - `search()`
  - `copySelectedOrFirst()`
  - `select(_:)`
- 搜索执行放到后台队列，完成后回主线程更新 `results`；只应用最新一次 query 的结果，避免快速输入时结果乱序回填。
- Launcher 每次显示时显式触发 `activate()`：
  - 清空 query
  - 加载默认结果
  - 默认选中第一项
  - 聚焦搜索框
- 用户选择命令时执行：
  - 复制到剪贴板
  - `markUsed(id:)`
  - 重新执行当前查询，让排序即时反映新的使用行为
- 窗口仍不在选择后自动关闭，保留当前 launcher 交互模型。

## 对外接口 / 类型调整

- `CommandItem.id` 从 `UUID` 改为 `String`。
- `CommandItem` 删除 `context` 字段，新增 `usageCount`、`lastUsedAt`。
- `StorageService.save(command:context:)` 保留 `context` 参数，但底层改为 SQLite 存储。
- `StorageService.commands` 与 `fetchAll()` 不再作为主接口，改为候选查询接口。
- 新增 `DatabaseManager`、`FuzzyMatcher`、`CommandScorer`、`SearchService`。
- `StorageService` 预留 `delete(id:)`，本版不接 UI。

## 测试计划

- 新增 SwiftPM `Tests` target，至少覆盖存储、评分、搜索和 ViewModel 关键行为。
- 存储测试：
  - 首次保存可写入 SQLite。
  - 完全相同命令重复保存不会插入第二条。
  - `markUsed(id:)` 会递增 `usage_count` 并更新 `last_used_at`。
  - `delete(id:)` 可删除记录。
- 持久化测试：
  - 用临时数据库文件重建服务实例后，历史记录仍能读出。
- 搜索测试：
  - `kctl -> kubectl`、`gco -> git checkout` 必须命中。
  - 非空 query 不会扫描并返回全量历史，只返回受 `LIMIT` 限制的候选结果。
  - 空 query 只展示默认候选集，不直接展示整库。
- 排序测试：
  - 高频命令因为 `log1p` 增长受限，不会对低频但强匹配的新命令形成永久压制。
  - 最近使用在 7 天窗口内可提升排序，超出窗口后影响衰减为 0。
  - 同分时排序稳定，不发生列表抖动。
- Launcher 测试：
  - 每次打开都清空 query、聚焦输入框、默认选中第一项。
  - 回车和点击都会复制并触发 `markUsed`。
  - 选择后重新搜索，结果顺序会反映新的使用数据。

## 假设与默认值

- v1.2 不做 FTS、拼写纠错、alias、标签和 normalized 去重。
- 数据库层统一限制候选集大小，默认 `LIMIT 200`。
- `source` 是上下文预留字段，当前允许为 `NULL`，不参与搜索排序。
- 当前阶段不提供删除 UI，只保留服务接口。
- SQLite 继续使用系统 `SQLite3`，不引入第三方 ORM 或封装库。
