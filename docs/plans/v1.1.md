# CommandHub V1.1 计划

## 概要

- 在 V1 MVP 基础上补齐 5 个高价值短板：更宽容的命令解析、稳定的全局快捷键、去重、受控的内存存储容量，以及更完整的 Launcher 键盘交互。
- 当前阶段继续保持本地优先的 macOS SwiftUI 应用形态，存储仍为内存模式。

## 关键变更

- 将 Parser 升级为“宽松排除”模型：只过滤空行、`#` 注释行和过短内容；像 `sudo kubectl get pods`、`cd project && npm run dev`、`echo "hello"` 这类命令整行保留。
- 在存储层加入相邻去重：如果新命令与当前最新一条完全相同，则直接忽略。
- 内存历史上限固定为 200 条，超出后丢弃最旧的数据。
- 用 `HotKey` 替换当前基于事件监听的伪全局快捷键方案，改为真正的全局快捷键实现。
- 补齐 Launcher 键盘交互：打开时自动聚焦搜索框，支持上下移动选中项，回车复制选中命令，ESC 关闭窗口。

## 对外接口 / 类型调整

- `CommandParser.isCommand(_:)` 从关键字前缀匹配改为宽松排除策略。
- `StorageService.save(command:context:)` 增加相邻去重和容量裁剪逻辑。
- `HotkeyService` 改为基于 `HotKey` 实现，不再使用 `NSEvent.addGlobalMonitorForEvents`。
- `LauncherViewModel` 与 `LauncherView` 增加选中状态和键盘事件处理能力。

## 测试计划

- 解析测试：`sudo kubectl get pods`、`cd project && npm run dev`、`echo "hello"` 都必须被识别并写入列表。
- 解析测试：空行、`#` 注释行、单字符行必须被忽略。
- 去重测试：连续多次复制同一命令，只保留一条最新记录。
- 容量测试：命令总数超过 200 条后，列表中只保留最新的 200 条。
- 快捷键测试：在不同应用切换场景下，`Command + Shift + V` 仍可稳定唤起窗口。
- Launcher 测试：窗口打开即聚焦输入框；上下键能切换选中项；回车复制；ESC 关闭。

## 假设

- 命令解析仍是启发式规则，不做语义级 shell 解析。
- 去重只处理相邻重复，不做全历史去重或提升排序。
- V1.1 的容量上限固定为 200 条。
- 当前阶段全局快捷键方案选用 `HotKey`。
- Launcher 体验目标是达到稳定、工具级的基础交互，不包含高级布局、分组结果或命令执行流程。
